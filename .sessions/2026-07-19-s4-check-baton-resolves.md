## What shipped

Added `src/engine/checks/check_baton_resolves.py` — an ADVISORY-only (warn, never exit-affecting) checker on the `posture="advisory"` seam that verifies every `## Next-2 baton` entry in each `control/status*.md` names a repo-relative path/anchor that still resolves on disk. It (a) locates the `## Next-2 baton` section (from its heading to the next `## ` heading or EOF); (b) extracts the inline code-span tokens that look like a repo-relative path — no whitespace, at least one `/`, a `.<ext>` file suffix on the last segment, optional `#anchor`; (c) resolves the path part against the target tree and, for a present anchor on a readable markdown file, resolves the anchor against the file's GitHub-slugified headings; (d) returns ONE advisory per unresolved reference, naming the token and its status file. A stale baton pointer — a renamed/deleted path still cited in the handoff — now surfaces instead of stranding the next wake on a dead file.

The token filter is conservative by design (fail-open by construction): version strings (`v1.19.0`), cron tokens (`2 */2 * * *`), CLI snippets (`check --strict`), and section names (`## Next-2 baton`) never match, so only an unambiguous repo-relative path that fails to resolve can ever fire. Input-gated + fail-open on every axis (no control dir / no status file / no baton section / unreadable file → silent); the anchor check itself fails open when the file is unreadable or carries no headings. Never raises; stdlib only. Multi-lane aware — globs `control/status*.md` so a repo's per-lane heartbeats (status-mining / status-exploration) are each graded. NOT in `STRICT_SUBCHECKS` (a stale baton pointer is a handoff nudge, not an adopter defect), wired exactly like the S3 `check_ungroomed_ideas` sibling.
