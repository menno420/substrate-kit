## What shipped

When `check` surfaces a finding it names the kind (the `[<kind>]` tag in the output) and describes the fix in prose; a host then has to re-derive the exact paste-ready form. S7 adds **`check --remediate <finding-kind>`**, a print-only CLI lookup wired on the same pre-check dispatch seam as R6's `check --explain-wall` (`_cmd_remediate`, always returns 0): given a finding kind, it prints the ready-to-apply remediation block directly.

**Contained/reversible by construction — the S7 guard rails hold with no file-rewriting path to make opt-in.** The spec (wave-2 doc S7) is "print paste-ready remediation blocks **only**", so this never became an auto-modify remediation. The command produces no findings, touches no files, and always exits 0 — even under `--strict`. A dedicated test (`test_cmd_check_remediate_is_print_only_modifies_nothing`) runs it against a populated tree under `--strict` and asserts every file is byte-identical afterward, so the opt-in/dry-run/no-mutation rails are enforced, not just claimed.

The seeded `REMEDIATIONS` registry is keyed by the **exact `Finding.kind` string** each checker emits (so `check --remediate <kind>` takes the same token `check` prints in `[<kind>]`) and covers the 6 kinds whose fix has a concrete paste-ready form:
- `folded-gate-mtime-picker` — reuses `check_folded_gate.REMEDIATION_SNIPPET` **verbatim** (imported, single source of truth — never duplicated; a test asserts `remediate(FINDING_KIND) == folded.REMEDIATION_SNIPPET`).
- `recipe-applies-when` — the `> **applies-when:** \`path:…, content:…\`` badge form.
- `stale-wall` — the dated `## Append log` re-verify row (or correct/delete a transient wall).
- `wall-ledger-disagree` — reconcile the `## Walls` correction vs the newest `## Append log` entry.
- `baton-unresolved` — retarget the `## Next-2 baton` token at a real path/anchor.
- `ungroomed-ideas` — run a groom pass + write a new groom doc (`/groom-ideas`).

An unknown/empty kind is not an error — the command lists the covered kinds and exits 0, so a host discovers the vocabulary instead of guessing.

**Dist:** the module ships in `dist/bootstrap.py` (CLI imports it), so it is in `MODULE_ORDER` **after** `check_folded_gate.py` (it imports that module's `FINDING_KIND`/`REMEDIATION_SNIPPET`). The intra-package import is **un-aliased on purpose** — the dist flattens every module into one namespace and strips these imports, so an aliased name would NameError in dist and trip `test_no_aliased_intra_package_imports_in_engine` (caught + fixed mid-session). Dist rebuilt via `python3 src/build_bootstrap.py` and byte-pinned (idempotent across two rebuilds; the 42 dist/pin tests pass).
