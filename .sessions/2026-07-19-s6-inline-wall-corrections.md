## What shipped

The engine `check_no_false_walls` leg is a REQUIRED-check false-wall gate: when a forward-binding adopter surface (`docs/**`, root `CONSTITUTION.md`/`CAPABILITIES.md`, live `.claude/**`) asserts a false agent-capability limitation ("the owner is the merge authority", "agents cannot delete branches", "merging is owner-gated"), it reds `check --strict`. Until this change the emitted `Finding` message carried only a generic blurb ("agents have no owner-imposed limitations; merging/deploying/pushing is normal agent work"). The SPECIFIC per-rule ground truth already existed as `WALL_CORRECTIONS` (a `dict[rule → correction sentence]`, added by R6/#497) but was reachable only through the separate `check --explain-wall <phrase>` lookup — a second manual step.

S6 inlines `WALL_CORRECTIONS.get(hit.rule, _UNKNOWN_RULE_CORRECTION)` directly into the `Finding` message, so the red gate now reads e.g.:

> `docs/x.md · false-wall:owner-is-merge-authority — line 1: false agent-capability wall 'owner is the merge authority' [owner-is-merge-authority] — The owner is not the merge authority: the only merge gate is CI green, and agents merge their own green PRs. Correct the line, or date-stamp/repudiate it if it records a genuine momentary refusal.`

The message now surfaces (a) the matched phrase, (b) the rule name `[<rule>]`, (c) the rule-specific capability truth, and (d) the correct/date-stamp instruction — in one shot, no second `--explain-wall` call.

**Behavior parity (required check) — preserved on every detection axis:** same `false-wall:<rule>` finding kind, same scanned paths, same exit-affecting-under-`--strict` loop. ONLY the human-readable `message` text changed. No existing test asserted the old message string (verified by grep across `tests/`), so the required-check contract is intact — a clean tree stays silent, every prior must-fail line still reds, and `test_explain_wall.py` (which owns the `WALL_CORRECTIONS` coverage guarantee) is untouched.
