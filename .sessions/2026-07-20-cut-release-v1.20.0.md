- **💡 Session idea:** **`cut_release.py --dispatch-release` (or a `check_release_ref_is_bump` preflight)** — this session's `verify_release.py` `is-the-bump` FAIL was structural, not accidental: a release cut inevitably lands doc-aftermath commits (CAPABILITIES append, current-state/status ledgers) on the branch AFTER the version-bump commit, and dispatching `release.yml` on the branch HEAD then tags a commit whose parent already declares the new version → the leg reds every time the aftermath is more than the bump. Two clean fixes: (a) teach `cut_release --write` to optionally capture the bump commit SHA and print/execute the exact `release.yml` dispatch **pinned to that SHA** (so the tag lands on the bump, not the branch HEAD); or (b) a lightweight `check_release_ref_is_bump` advisory that, given a target version, warns when the ref you're about to dispatch has a parent already declaring that version. Deduped against `docs/ideas/` (grep `release`/`tag`/`bump`/`cut` — the three hits are changelog-`[Unreleased]`-structure, staged-artifact-regen-lag, and upgrade-checklist-release-json-placement; none address release-ref-vs-bump placement). Worth having because it converts a recurring, silent release-provenance red into a caught-at-cut decision — the same "friction → guard" instinct the whole wave-2 ladder rewards.
