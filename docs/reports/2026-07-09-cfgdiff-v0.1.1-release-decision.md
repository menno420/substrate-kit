At wind-down (2026-07-09) the sonnet5 arm held cfgdiff 0.1.1 on `main` —
independently verified, 165 tests + 4 deliberate xfails, CI green across 3
Python versions — with **zero tags on origin** and release automation armed
but never fired. This writeup records the release decision the lane made,
its reasoning, and the exact owner path it flagged. It is the release-writeup
convention in miniature: the decision, the alternative, the walls with exact
error text, and a paste-ready owner step — all committed before wind-down.

## The decision (wind-down review §(ii), verbatim)

> **Decide-and-flag.** "Published" was ambiguous → shipped
> installable-from-git, flagged PyPI as ⚑ owner. v0.1.0-tag-or-not →
> recommended releasing 0.1.1 directly, flagged the reasoning. No decision
> waited on a human that didn't strictly need one.

Concretely: skip tagging v0.1.0 and release **0.1.1 directly** — 0.1.1 is
the version that contains both the release automation and the three
differential-corpus bug fixes (see
[the differential-testing method writeup](2026-07-09-cfgdiff-differential-testing-method.md)),
and the v0.1.0 commit predates the release workflow entirely.

## The flagged owner path (final `control/status.md`, verbatim)

> 2. Release 0.1.1 with one push:
>    `git tag -a v0.1.1 0b1eb60 -m "cfgdiff 0.1.1"` &&
>    `git push origin v0.1.1`
>    — runs the test gate, builds sdist+wheel, cuts the GitHub release with
>    the CHANGELOG 0.1.1 section as body, and (if item 1 is done first)
>    publishes to PyPI. Do NOT tag v0.1.0 at 0260aae expecting automation —
>    it predates release.yml and would fire nothing.

Item 1 (the 2-minute prerequisite, biggest payoff): register the PyPI
trusted publisher — pypi.org → Publishing → "Add a pending publisher" with
owner `menno420`, repository `codetool-lab-sonnet5`, workflow `release.yml`,
environment `pypi`.

## The walls that shaped the decision (NEXT-BOOT § known walls)

The release had to be structured as *agent merges everything, owner pushes
one tag* because every direct path was probed and denied with captured
errors (the source's rule: "probing a documented wall twice is a bug"):

- Tag push and branch delete both fail identically at the git proxy:
  `error: RPC failed; HTTP 403 curl 22 The requested URL returned error: 403`.
- The github MCP exposes **no release/tag creation tools** — only read-only
  ones (list_releases, get_latest_release, get_release_by_tag, get_tag,
  list_tags); confirmed by enumerating the server's toolset.
- Direct api.github.com is blocked: "GitHub access is not enabled for this
  session".
- PyPI publishing needs the owner-side trusted-publisher registration; after
  that, `.github/workflows/release.yml` handles everything on a tag push via
  OIDC trusted publishing — "no session credential exists or is needed."

So the release path (source's proposed instructions, verbatim): "you merge
version+CHANGELOG; the owner pushes the tag; `release.yml` does the rest via
OIDC trusted publishing — no session ever holds a credential."

## Honest residue (recorded at wind-down)

- `release.yml` had **never actually fired** at wind-down (the tag push is
  owner-only), so its first real run is the owner's v0.1.1 push itself.
- The tag/release remained un-cut as of the 2026-07-12 consolidation census
  ("zero tags exist on origin") — the owner's release-then-archive sequencing
  for the sonnet5 repo (consolidation plan Phase 2 decision 1 / Phase 3.2)
  depends on pushing the tag **before** the archive click, since archiving
  freezes the tag-push path.
