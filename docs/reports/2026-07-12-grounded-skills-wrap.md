## 2. What the owner's five ideas became

The plan (§1) consolidated the owner's directive into five ideas, rooted in
two failure causes: agents don't reliably know their own capabilities, and
there was no centralized playbook set for recurring actions. What each became:

**1. Capability self-knowledge → the CAPABILITIES ledger (slice 5, #274).**
Every adopter repo now carries a `docs/CAPABILITIES.md` that is generated and
refreshed, not hand-drifted: a kit-owned marker-fenced seed section that
upgrades refresh automatically, a venue column + posture rule (which session
venue can do what), per-entry dates with a staleness clause, and an advisory
checker. An agent waking in a fleet repo this evening reads a verified
can/cannot ledger instead of guessing at walls — and the discovery rule tells
it exactly how to add a new finding the same session.

**2. The skill layer → the SKILLS list + generated docs/SKILLS.md index
(slices 1–2, #264/#265).** Recurring actions (session close, upgrade
distribution, release cuts, and the rest) are now kit-owned skills with
playbook-grade bodies — exact commands, exact file targets, exact links, a
structured `grounds` field per skill — indexed in a generated `docs/SKILLS.md`
wired into every adopter's boot set. An agent no longer improvises a procedure
for a recurring action; it opens the index and runs the grounded playbook.

**3. The owner-assist standard → control-README + collaboration templates
(slice 4, #272).** Anything routed to the owner must now be paste-ready: the
OWNER-ACTION field set (what/where/how/why/unblocks/verified-needed) extended
into the control README and collaboration-model templates, with an advisory
checker flagging vague destinations and missing risk classes. The owner never
derives anything — every ask arrives as a click path plus paste-ready text.

**4. The intake playbook → the /intake skill (slice 3, #270).** The
understand-and-reflect doctrine (superbot Q-0254) is now executable: a skill
that consolidates the owner's fragmented asks into their few main ideas, maps
them to known step patterns, and states the fuller picture back before
building. What was a prose rule agents had to remember is now a runnable
procedure.

**5. Centralized + self-propagating → index wiring + seat-digest render +
registration reflex (slices 1/6/8: #279 + fm #126, #282).** The kit is the
single home: the boot-set wiring makes the index reachable in every adopter;
the seat-digest render (kit #279) exposes a machine extraction contract that
fleet-manager's prompt tooling now consumes (fm #126 @ 434c5f9) — one source,
zero hand-copied third copies; and the CONSTITUTION registration-reflex clause
(#282) teaches agents to grow the skill set themselves (workflow → index row →
promoted skill) while keeping the propose-don't-apply boundary intact. What an
agent in a fleet repo has tonight that it didn't have this morning: a boot set
that routes to a verified capability ledger and a grounded skill index, owner
asks that arrive paste-ready, an executable intake procedure, and a doctrine
that makes the system extend itself.
